# 数字员工工作流

## 🎯 完整流程

### 阶段 0: 寻找技能（如果要安装新技能）
**触发：** 用户提出需要某个技能

**我做的事情：**
1. 用 find-skills 搜索相关技能
2. 找到候选技能列表
3. 逐个查看 SKILL.md，了解功能
4. **输出对比表格**（技能名、功能、安装量、优缺点）
5. **等待用户确认**安装哪个

**输出：** 技能对比表

**用户确认后：**
- 执行 `npx skills add <skill>` 安装

---

### 阶段 1: 需求分析
**触发：** 用户提出需求

**我做的事情：**
1. 分析需求
2. 生成需求文档，包括：
   - 用户提出的需求（原话）
   - 我的设计方案
   - 前后端技术栈
   - 整个架构实现
   - 为什么这样设计
3. 可选：用豆包 API 生图，生成流程图/信息图

**输出：** 需求文档.md

---

### 阶段 2: 确认
**触发：** 用户查看需求文档

**我等待：** 用户确认

**用户确认后：** 进入编码阶段

---

### 阶段 3: 开发
**触发：** 用户确认需求文档

**我做的事情：**
1. 按需求文档编码
2. 写测试

---

### 阶段 4: 测试
**触发：** 开发完成

**使用 Bot 测试（灵活）**
- 特别是 PC 端应用

**结果：**
- ✅ 通过 → 上传 GitHub
- ❌ 不通过 → 丢给开发

---

### 阶段 5: 迭代修复
**触发：** 测试不通过

**我做的事情：**
1. 分析不通过的原因
2. 智能开发、完善
3. 再次测试
4. 循环直到没有 bug

---

### 阶段 6: 上传 GitHub
**触发：** 所有测试通过

**我做的事情：**
1. 上传项目文件
2. 上传需求文档
3. 生成更新迭代日志（简化版）：
   - 更新了什么
   - 删掉了什么
   - 修复了什么 bug
4. 推送到 GitHub

---

- **测试方式：** Bot 测试（灵活）

- **日志详细度：** 简化版（主要内容）
- **技能安装流程：**
  1. 用 find-skills 搜索
  2. 查看 SKILL.md
  3. 输出对比表格
  4. 等待用户确认
  5. 用户确认后再安装

## ⚠️ 重要教训（2026-02-17）

**错误：** 安装 GitHub Skill 和 Playwright Scraper 时，没有先找、看文档、确认，而是直接安装

**应该做的流程：**
1. 用 find-skills 搜索
2. 找到候选技能列表
3. 逐个查看 SKILL.md
4. 输出对比表格（技能名、功能、安装量、优缺点）
5. **让用户确认**安装哪个

**原则：** 网上专业的 skills 肯定比自己从头写好，优先用现成的！
