# Recurring Patterns - 重复模式追踪

> 识别重复的任务和请求，找到自动化机会

---

## 2026-02-17

### 1. X 文章处理
- **频率:** 中等（3-5 次/周）
- **流程:**
  1. 用户提供 X 文章链接
  2. 获取文章内容（使用 jina.ai API）
  3. 总结核心观点和关键信息
  4. （可选）写入 Obsidian vault
- **用户偏好:** 默认只总结，明确要求才写入 Obsidian
- **自动化机会:**
  - 检测到 X 链接时自动识别文章类型
  - 根据内容类型智能建议是否写入 Obsidian
  - 创建 X 文章归档工作流

### 2. 技能探索与安装
- **频率:** 低-中等（1-2 次/周）
- **流程:**
  1. 用户问"有什么技能"或"能做 X 吗"
  2. 我搜索 skills.sh
  3. 安装和配置
  4. 测试和验证
- **用户偏好:**
  - 优先搜索网上已有的成熟 skills，而不是从头写
  - 查看 SKILL.md，不能只看名字猜功能
  - 创建文件前必须先说明要做什么
- **自动化机会:**
  - 定期检查新技能并主动推荐
  - 建立技能评估模板（功能测试、适用场景、优缺点）

### 3. 文件和仓库管理
- **频率:** 中等（每次新项目）
- **流程:**
  1. 创建项目文件夹
  2. 初始化 Git 仓库
  3. 创建 GitHub 仓库
  4. 推送代码
- **用户偏好:** 用户偏好 proactive action，"下次这种事情不要问我"
- **自动化机会:**
  - 创建一键初始化脚本（文件夹 + Git + GitHub）
  - 标准化项目结构模板

### 4. 日常检查和总结
- **频率:** 每日（daily-heartbeat cron）
- **流程:**
  1. 读取 HEARTBEAT.md
  2. 检查追踪文件（proactive-tracker, recurring-patterns, outcome-journal）
  3. 检查上下文 %
  4. 更新 MEMORY.md
  5. 思考主动改进机会
- **自动化机会:** 已通过 cron 实现 ✅

### 5. 代码测试和验证
- **频率:** 不定期
- **流程:**
  1. Agent 生成代码
  2. 我进行代码审查
  3. 生成测试报告
  4. 发现问题和改进建议
- **用户反馈:** "测试怎么测试呢？你是依靠代码来嘛？要是你的代码逻辑也是有问题呢？能替代吗？"
- **问题:** WSL 环境无法直接控制 Windows 浏览器
- **自动化机会:**
  - 部署到线上环境（GitHub Pages/Vercel）
  - 使用 E2E 测试框架（Playwright/Cypress）
  - 集成到 CI/CD 流程
